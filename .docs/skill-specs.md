# Skill Specifications

> Each skill below is defined with enough context for a Claude session to implement the SKILL.md file. Every aspect includes: the **chosen behavior**, **source framework** it comes from, and **why** it was chosen for a multi-platform team context.
>
> For the 9-phase comparison tables these picks were derived from, see [compare.md](compare.md).

## 1. `/letsgo` — Brainstorming & Design

**Purpose**: Ensure every piece of work has a shared understanding between developer and AI before any code is written. Right-sizes ceremony to avoid both "skipping design" and "over-engineering a one-liner."

**Fork from:** CE's `ce:brainstorm` (modify to match spec below)

| Aspect | Chosen Behavior | Source | Why |
|--------|----------------|--------|-----|
| **Invocation** | Manual activation via `/letsgo`. User-facing, never auto-triggered. | CE | Developers must consciously decide to start design. Auto-routing (CC10X) removes agency. |
| **Core philosophy** | "No code without confirmed purpose and scope" — but right-size the depth. Even a trivial change must have Purpose + Scope stated out loud. A complex feature gets the full design treatment. | CE pragmatism + Superpowers floor | CE prevents ceremony overload. Superpowers' hard floor prevents skipping entirely. The hybrid: you always do *something*, but how much depends on scope. |
| **Pre-dispatch assessment** | Before dispatching research agents, evaluate: "Can I construct a useful search scope from this request?" If the request is ambiguous, ask the user one targeted question with an explanation of why: "I need to know where to focus the research because [reason]. [Specific question]." Only dispatch agents after the scope is clear. | Custom | Prevents agents from running blind searches on vague requests. The user provides direction, agents execute with clear scope. Uncertainty resolution happens *before* dispatch, not during. |
| **Research** | Dispatch **2 parallel agents** from `.claude/agents/` for context preparation: **(1)** `code-explorer` — scans relevant code paths, module READMEs, git history. Excluded: `.docs/`. **(2)** `docs-explorer` — scans `.docs/work/` (brainstorms, retros with past knowledge), project docs, CLAUDE.md, AGENTS.md. If internal docs lack answers, escalates to external sources (context7 MCP, web search). Excluded: source code, in-progress work items. Both agents always dispatch regardless of scope tier — you need context to classify scope accurately. | Custom (inspired by CE's research agents) | In multi-platform projects the codebase can span multiple repositories or modules. Without upfront context gathering, the AI asks questions it could answer itself by reading the code. Parallel agents are fast and cheap. Two agents with clean separation (code vs. knowledge) prevent overlap and keep each focused. |
| **Scope assessment** | After agents return, classify the request as **lightweight** (< 30 min, single file/function), **standard** (hours, multi-file), or **deep** (days, multi-subsystem). AI proposes classification, **user confirms or overrides**. For deep: decompose into sub-projects. The classification determines how much ceremony follows. | CE classification + Superpowers decomposition | CE's 3-tier classification is the simplest routing mechanism. Superpowers' decomposition catches projects that look standard but are actually deep. User confirmation prevents misclassification. |
| **Question style** | One section at a time, following a **fixed 6-section progression**: (1) Purpose & Context — informed by agent findings, confirm/correct. (2) Scope & Boundaries — what's in, what's out, plus scope classification. (3) Affected Systems — which modules, services, or layers are touched. (4) Approaches — 2-3 with pros/cons, anti-anchored (all shown THEN recommend). Lightweight: 1 approach with "any concerns?" (5) Key Constraints & Risks — technical limits, third-party dependencies, breaking changes. (6) Success Criteria — how do we know it's done. Each section: present findings/recommendation → user confirms or corrects → lock section → next. | CE + Superpowers incremental validation | Section-by-section prevents overwhelm. 6 defined sections ensure consistent coverage. Using research results as defaults respects the developer's time — they confirm rather than explain. Anti-anchoring on approaches is a hard rule. |
| **Approaches** | Present 2-3 approaches with pros/cons. Show ALL approaches FIRST, then recommend one (anti-anchoring is a hard rule). Never lead with the recommendation. For lightweight scope: may present just 1 approach with "here's what I'd do, any concerns?" Approaches are always at the **architecture level** — per-platform implementation details are deferred to `/plan`. | CE | Anti-anchoring is critical when the AI has a strong prior. Architecture-level approaches prevent premature platform-specific decisions. Per-platform details belong in planning, not brainstorming. |
| **Self-review** | Before presenting to user for final approval, run an **inline self-review** (not a separate agent file — the review criteria are tightly coupled to the brainstorm format). Checks: (1) No placeholders ("TBD", "TODO", "etc.", "similar"). (2) Consistency — do later sections contradict earlier ones? (3) Scope — does the design actually address the stated purpose? (4) Completeness — are all affected systems addressed? **Runs on all tiers**, adapted to format: for standard/deep, reviews the document; for lightweight, reviews the conversational decisions. The self-review receives enough context to make a meaningful assessment regardless of format. | CE subagent review + Superpowers checklist | The main agent is biased toward its own design. The 4-check list catches inconsistencies. Running on all tiers (including lightweight) prevents even small changes from having unvalidated assumptions. |
| **Output artifact** | Write to **`.docs/work/YYMMDD-<slug>/brainstorm.md`**. Folder is created eagerly — brainstorm.md is persisted immediately even if planning hasn't started. For lightweight scope: skip the full document, but Purpose + Scope + Approach must still be confirmed verbally (this IS the record). For standard/deep: full document with the 6 sections. | CE (modified) | Co-locating brainstorm.md with plan.md under the same slug creates natural traceability. Eager folder creation means "I'll plan later" leaves a persistent reminder. The brainstorm.md represents the **current state of decisions**, not the history of exploration. |
| **Handoff** | **Opinionated presentation** — recommend the next step based on scope tier. For standard/deep: "I recommend proceeding to `/plan` now while context is fresh." Options: (1) "Proceed to planning" → invoke `/plan`. (2) "I'll plan later" → save brainstorm, done. (3) "Execute directly" → invoke `/worker` (future, only for lightweight — unavailable until `/worker` is built). | CE flexible handoff | Opinionated handoff saves a round-trip — the right answer is obvious 90% of the time. Option 3 routes to `/worker` (not raw implementation) to maintain workflow discipline even for small changes. |
| **Visual companion** | Opt-in only. If the user asks for visuals, produce ASCII diagrams in the conversation (not in the document). Never auto-generate mockups. | Superpowers (stripped down) | ASCII is readable in any terminal. Browser mockups (Superpowers' default) are heavyweight and don't render in all environments. The user can ask if they want them. |
| **Codebase verification** | Any claim about what exists or doesn't exist in the codebase MUST be verified by actually reading the code. "I don't think we have a payment module" is not acceptable — check first. | CE | "Claims of absence must be verified" — this catches the most common brainstorming error: designing something that already exists, or assuming an integration point exists when it doesn't. |

## 2. `/plan` — Planning

**Purpose**: Transform a brainstorm into an executable plan with CE-style implementation units that an AI agent can execute with TDD discipline. The plan is the contract between the designer (human) and the builder (AI agent or human).

**Fork from:** CE's `ce:plan` (modify to match spec below)

| Aspect | Chosen Behavior | Source | Why |
|--------|----------------|--------|-----|
| **Trigger** | Always requires a `brainstorm.md` as input. Uses **CE-style artifact discovery**: (1) If argument provided (`/plan 250410-stripe-webhooks`) → use directly. (2) If no argument → scan `.docs/work/*/` for folders with `brainstorm.md` but no `plan.md`. (3) One match → propose it. (4) Multiple → ask user. (5) None → tell user to run `/letsgo` first. Can also resume/deepen an existing plan. | CE artifact discovery | Session-independent handoff — survives `/compact`, new sessions, "I'll plan later." No reliance on conversation context. Mandatory brainstorm input ensures no plan is created without shared understanding. |
| **Implementation units** | **CE-style implementation units** — logical, testable chunks of work. Each unit is one coherent behavior that can be tested independently. Units follow the **vertical slice** principle — each produces something testable end-to-end, not a horizontal layer. Size varies (some 5 min, some 20 min) but each is **one behavior, one test boundary**. TDD discipline operates *within* each unit at execution time. | CE + Matt Pocock | CE's units are the natural fit for behavioral plans. The implementing agent reads the unit's goal + test scenarios and applies TDD internally. Vertical slicing catches integration issues early. |
| **Unit structure** | Each implementation unit contains: **Goal** (behavioral — what it accomplishes), **Dependencies** (which prior units must complete), **Confidence** (`[GREEN]` / `[YELLOW]` / `[RED]` — ASCII, no emoji), **Affected Systems** (which modules, services, or layers are touched), **Approach** (optional — only when non-obvious or to prevent wrong paths; omitted when goal + test scenarios say it all), **Test Scenarios** (concrete scenarios to verify), **Verification** (how to confirm the unit is done). | CE + CC10X + Matt Pocock | Behavioral descriptions trust the implementing agent. Optional approach field prevents wrong paths without over-constraining. Traffic light confidence is simpler than numerical scoring with no false precision. ASCII for universal compatibility. |
| **Code in plans** | **No code, no pseudo-code, no `file:line` references.** Plans describe *what* to build, not *how*. The implementing agent reads the codebase and discovers patterns itself. Trust the agent's capability, constrain it with decisions. | CE + Matt Pocock | Full code goes stale. File references break on refactor. The implementing agent has the codebase in front of it — it doesn't need the plan to duplicate that information. Plans that describe behavior precisely give the agent freedom to implement correctly for the current codebase state. |
| **Durable decisions** | **Top-level section** (above implementation units) for any plan touching multiple files or systems. Each decision includes **both behavioral contract AND schema-level precision**: the behavioral contract explains what it does, who consumes it, what invariants hold; the schema precision specifies exact routes, response shapes, status codes. Covers: API routes/endpoints, database schema changes, shared type definitions, auth/authz boundaries, service boundaries, cross-platform contracts. | Matt Pocock (what) + CE (rationale) | Durable decisions are the contracts between independently-built systems. Behavioral alone leaves room for "backend returns `{ok: true}` but frontend expects `{received: true}`." Schema precision eliminates ambiguity at integration boundaries. Both layers together explain *why* and specify *what exactly*. |
| **Placeholders** | **Hard ban** (auto-rejected): "TBD", "TODO", "add validation", "implement later", "etc.", "similar", "and so on", "as needed". **Soft ban** (flagged in review, author can justify): "appropriate", "relevant", "necessary", "proper", "handle accordingly", "standard", "as described above". Principle: if an implementing agent reading the phrase would need to make a judgment call about what it means, it's a placeholder. | Superpowers + CE | Hard ban catches obvious placeholders. Soft ban catches weasel words that feel specific but mean nothing to an implementing agent. The two-tier system prevents both lazy placeholders and false-precision prose. |
| **Dependency tracking** | **DAG rule**: Unit N may only depend on units that come before it. Cross-phase dependencies (e.g., "mobile depends on backend API being deployed") must be declared as explicit contracts with the interface specified. Circular dependencies cause plan review to fail. | CC10X | Explicit DAG prevents the "I built the mobile feature but the API isn't deployed yet" problem. Cross-phase contracts make handoff points visible. |
| **Plan review gate** | **Fail-closed.** Before the plan is presented to the user, it must pass **4 checks**: (1) **Dependency coherence** (inline) — does each unit's dependencies exist in the codebase or in a prior unit? No circular dependencies? No assumptions about capabilities that don't exist? (2) **Completeness** (via **subagent** — fresh eyes) — does the plan cover everything in the brainstorm's scope? (3) **No placeholders** (inline) — scan for hard-banned and soft-banned strings. (4) **Testability** (inline) — does every unit have at least one concrete test scenario? Retry up to 3 times on failure. After 3 failures, surface issues to user. | CC10X + Custom | 4 checks instead of 3: added testability because a unit without test scenarios can't be verified. Completeness uses a subagent because comparing plan vs. brainstorm requires judgment (the planning agent is biased). Mechanical checks (1, 3, 4) are inline. |
| **Confidence** | **Traffic light system**: `[GREEN]` (clear, ready to execute), `[YELLOW]` (some ambiguity, implementable), `[RED]` (blocked, needs clarification). No `[RED]` units allowed in a finalized plan — they must be resolved first. Overall plan confidence = lowest unit confidence. | CC10X (simplified) | Traffic light communicates the right information without false precision of 0-100 scores. GREEN = go, YELLOW = agent may need judgment, RED = stop. Simpler, more honest. |
| **Research integration** | If planning surfaces new unknowns, dispatch the **`docs-explorer` agent** (same agent used by `/letsgo`) with a targeted question. The agent checks `.docs/work/*/retro.md` for past knowledge first, escalates to external sources if needed. | CE (adapted) | Reusing the same docs-explorer agent prevents divergent research mechanisms. One agent, different prompts — broad context during brainstorming, specific questions during planning. |
| **Resume/deepen** | If a plan already exists for this work, resume from where it was. If the user says "deepen" for a specific section, re-assess that section's confidence and expand. Deepening triggers a confidence re-check on the whole plan. | CE | Plans evolve. Being able to say "deepen the auth section" without restarting saves significant time. |
| **Output location** | Write to **`.docs/work/YYMMDD-<slug>/plan.md`** alongside the brainstorm.md. | CE (modified) | Co-located with brainstorm under the same slug. The folder is the unit of work — everything related lives together. |
| **Plan document structure** | Top-level structure: **Context** (source brainstorm, scope, affected systems) → **Durable Decisions** (behavioral + schema precision) → **Risk Assessment** (standard/deep only) → **Implementation Units** → **Overall Confidence**. For deep scope with many units: include ASCII DAG visualization at the top. | CE + CC10X + Custom | Durable decisions sit above units so any unit can reference them. Risk assessment before units gives context. ASCII DAG for deep plans gives the full picture at a glance. |
| **Execution handoff** | After plan approval, present options: (1) "Execute now" → proceed to `/worker` (future). (2) "Execute in worktree" → create worktree first, then execute (future). (3) "I'll execute later" → save plan, done. (4) "Assign to team" → format as issues (future capability). | CE flexible handoff | Teams have both humans and AI agents executing plans. The handoff must accommodate both. Worktree option is critical for parallel development. |
| **Risk assessment** | **Mandatory for standard and deep scope** (skipped for lightweight). Structured table: risk description, probability (low/medium/high), impact (low/medium/high), mitigation strategy. Must cover at minimum: data loss scenarios, breaking changes to existing APIs, cross-platform compatibility risks, third-party service dependencies. | CC10X | For teams shipping fast, unassessed risks become production incidents. The mandatory categories cover common risk areas — API breaking changes can affect multiple consumers (e.g., mobile clients, frontend apps, backend services), and third-party dependencies are a significant external risk. |

## 3. TDD — Test-Driven Development (Internal Skill)

**Purpose**: Enforce test-first development with behavioral testing through public interfaces. Every feature and bug fix follows RED → GREEN → REFACTOR with process discipline that prevents common shortcuts.

**Invocation**: Internal skill — referenced by implementing skills/agents, never user-invoked directly. There is no `/tdd` command. TDD is how all implementation happens.

**Fork from:** Superpowers' TDD as core base, layer in Matt Pocock (vertical slicing, test behavior through public interfaces). Platform files forked from ECC.

**File structure**: Core discipline file + per-platform files. The calling skill/agent injects core + the relevant platform file based on which platform the task targets.
- `.claude/skills/tdd/SKILL.md` — universal discipline (RED-GREEN-REFACTOR, rationalizations, mocking rules, delete policy)
- `.claude/skills/tdd/<platform>.md` — one per platform/language in the project's tech stack, forked from ECC's platform-specific testing skills

| Aspect | Chosen Behavior | Source | Why |
|--------|----------------|--------|-----|
| **Iron law** | "No production code without a failing test first." Tests verify behavior through public interfaces, not implementation details. Always use run mode, never watch mode. | Matt Pocock (philosophy) + CC10X (process) | Matt Pocock's "test behavior through public interfaces" prevents brittle tests that break on refactor. CC10X's watch mode ban prevents silently swallowed failures — watch mode re-runs on save and can mask flaky tests or partial passes. |
| **Slicing** | **Vertical tracer bullets.** The first test for any feature proves the full path end-to-end (e.g., API request → handler → DB → response shape). Subsequent tests fill in edge cases and error paths. Never build horizontal layers (all models, then all routes, then all handlers). | Matt Pocock | In multi-platform projects, a tracer bullet test that hits the backend → API response → validates the shape that cross-platform consumers will use proves the contract before building out either side. This catches integration issues at the start, not the end. |
| **Red-Green-Refactor cycle** | **(1) RED**: Write the test. Run it. It MUST fail. If it passes immediately, you're testing existing behavior — delete the test and rethink. Distinguish between runtime RED (assertion fails) and compile-time RED (won't compile) — both are valid but mean different things. **(2) GREEN**: Write the minimum production code to make the test pass. No more. Run the test again — it must pass. **(3) REFACTOR**: Only after GREEN. Clean up both test and production code. Run all tests after refactoring — everything must still pass. **(4) CHECKPOINT**: Create a git commit at each phase transition. Commit messages: `test(red): <what the test expects>`, `feat(green): <what was implemented>`, `refactor: <what was cleaned up>`. **Squash before PR** — checkpoints are for rollback safety during development, not permanent history. | CC10X (checkpoints) + Matt Pocock (never refactor while RED) | Git checkpoint commits give rollback safety — if refactoring breaks something, you can reset to the GREEN commit instead of debugging your refactor. The commit message convention makes the TDD history visible in git log. Squashing before PR keeps the final history clean. |
| **Coverage target** | **80%+ minimum** across statements, branches, functions, and lines. Not 100% — that incentivizes testing trivial code (getters, type guards) and leads to a false sense of security. Focus coverage on business logic and boundary code. | CC10X + ECC | 80% is the industry sweet spot. Below 80%, you're likely missing important paths. Above 90%, you're likely testing implementation details. In practice: backend business logic and API handlers should be near 100%, UI components can be lower. |
| **Mocking rules** | **Boundary-only mocking.** Only mock at system boundaries: external APIs, third-party services, databases, caches, time, filesystem, network. Use dependency injection to make boundaries mockable. Within the application boundary, use real code — no mocking internal modules. Use factory patterns for test data (not inline object literals). | Matt Pocock (DI) + CC10X (factory pattern) | Projects with external dependencies (payment providers, auth services, analytics, push notifications, etc.) must mock at these boundaries. Mocking internal modules is not necessary — it couples tests to implementation. Factory patterns keep test data consistent and maintainable across the test suite. |
| **Rationalization prevention** | Embed the following **12 red flags** that signal TDD is being bypassed. If ANY are detected, stop and correct before continuing: (1) "I'll write the test after" (2) "This is too simple to test" (3) "I'll just manually verify" (4) "The test is obvious so I'll skip RED" (5) "I need to see the implementation first" (6) "Let me just get it working then add tests" (7) "This is just a refactor, no tests needed" (8) "The existing tests cover this" (without verifying) (9) "I'll add tests in a follow-up" (10) "This is a prototype/POC" (11) "The test would be too hard to write" (12) "Watch mode shows it's passing" (watch mode is banned). | Superpowers | These 12 rationalizations are the most common ways TDD gets skipped in practice. Making them explicit and naming them means the AI agent (or human developer) can catch themselves. Superpowers' research on psychological drift in AI agents showed these patterns emerge reliably. |
| **Failure handling** | If a test passes immediately after writing it (before implementation): the test is wrong — it's testing existing behavior or is vacuous. Delete it and write a test that actually exercises the new behavior. If a test fails at compile time (not runtime): this is valid RED for type-driven development — the type system is telling you the interface doesn't exist yet. Proceed to GREEN by creating the interface. | CC10X | The runtime-vs-compile-time distinction matters heavily in languages with strong type systems. A type error is a different signal than an assertion failure — both are RED, but they tell you different things about what to implement next. |
| **Loop cap** | **"Never refactor while RED"** — if tests are failing, you're in GREEN mode (make them pass), not REFACTOR mode. If you've attempted 3 fixes for the same failing test without success, stop — treat it as an architectural problem and escalate to the user or invoke the debugging skill. | Matt Pocock + Superpowers | 3 failed attempts means you're likely misunderstanding the problem, not making typos. Escalating early prevents the sunk-cost spiral where you keep hacking at a test for 30 minutes. |
| **Delete policy** | Code written before its test MUST be deleted entirely and rewritten test-first. No exceptions. "But it works" is not an argument — the point of TDD is that tests drive design. Pre-written code was designed without test feedback. | Superpowers + CC10X | This sounds extreme but it's the only way to maintain TDD integrity. If pre-written code stays, you're retro-fitting tests (verification) not doing TDD (design). The code you rewrite will be better because the test told you what the interface should look like. |
| **Anti-patterns reference** | Maintain awareness of these test anti-patterns: (1) Testing implementation details (private methods, internal state). (2) Tautological tests (test restates the implementation). (3) Flaky tests (time-dependent, order-dependent, network-dependent). (4) God tests (one test validates 10 behaviors). (5) Mock-heavy tests (mocking internal modules). (6) Snapshot overuse (snapshots of large objects that change frequently). | CC10X + Superpowers + Matt Pocock | Having a named list of anti-patterns means reviews can reference them by name ("this is a tautological test") rather than explaining from scratch each time. |
| **Platform-specific** | Separate files forked from ECC, injected by the calling skill based on platform. Each platform file covers the appropriate test runner, mocking library, and coverage tool for that language/framework. | ECC (forked and trimmed) | Each platform has different testing idioms and frameworks. Separate files prevent injecting irrelevant platform guidance. Forked from ECC's battle-tested, comprehensive testing skills. |
