## Context

Receives:
- Dispatch summary from start
- Explorer findings (codebase analysis)
- User input (requirements or problem description)
- Trace context (FIX mode only: reproduction, hypothesis, severity)

Reads:
- Source code referenced by explorers
- `.docs/research/` for prior research context
- Existing `.docs/sketches/` for pattern reference

## Procedure

### 1. Determine mode

- BUILD: new feature, enhancement, greenfield work
- FIX: from diagnose phase (bug, regression, incident)

### 2. Explore before drafting

Gate: **EXPLORE-BEFORE-IMPLEMENT** -- before writing any sketch content, verify understanding by reading actual source.

- Read files referenced by explorer findings
- Identify affected systems, entry points, data flow
- Note existing patterns, conventions, test structure
- If a question is answerable by exploring the codebase, explore instead of asking

### 3. Checkpoint summary loop

Repeat until mutual understanding is reached:

**A. Form assumptions from evidence**
- Read code, tests, configs. Extract facts.
- Form assumptions where evidence is partial.

**B. Present checkpoint**
- Facts discovered (with file:line references)
- Assumptions formed (clearly labeled)
- 0-2 genuine questions (with recommended answers)

**C. Anti-anchoring rule**
- For judgments (approach choices, priorities, tradeoffs): ask user's instinct BEFORE presenting your recommendation.
  Example: "Before I share approaches, what's your instinct on build-vs-buy here?"
- For facts (codebase state, test results, behavior): present finding, then confirm.

**D. User responds**
- Corrects assumptions, answers questions.
- Incorporate corrections, proceed to next round or finalize.

### 4. Draft sketch

Gate: **DESIGN-THEN-CODE** -- full sketch before any implementation thinking.

**BUILD mode sections** (see references/sketch-template.md):
1. Problem / Opportunity
2. Motivation (why now)
3. Current State (what exists today)
4. Affected Systems
5. Approaches Considered (2-3 minimum, user instinct first)
6. Constraints and Non-Goals
7. Success Criteria
8. Research Context
9. Open Questions (blocking vs deferred)

**FIX mode sections** (see references/sketch-template.md):
1. Symptoms Observed
2. Reproduction Steps (from diagnose)
3. Root Cause Hypothesis
4. Blast Radius
5. Affected Systems
6. Constraints
7. Research Context
8. Open Questions (blocking vs deferred)

### 5. Approaches (BUILD mode)

Gate: **OPTIONS-THEN-RECOMMEND**

1. Ask user's instinct first (anti-anchoring)
2. Present ALL viable approaches (minimum 2, typically 3)
3. Per approach: summary, pros, cons, effort estimate, risk
4. THEN recommend with rationale
5. User selects or modifies

### 6. Resolve blocking questions

- Blocking questions must be answered before passing to blueprint
- Deferred questions get logged but don't block
- If user cannot answer a blocking question, note the uncertainty and its implications

### 7. Write sketch.md

Gate: **ARTIFACT-BEFORE-HANDOFF** -- persist to disk before next phase.

Write to `.docs/sketches/<slug>.md` using references/sketch-template.md.
Slug was already generated by yo.

### 8. Confirm with user

Present sketch summary. User confirms or requests changes. Iterate if needed.
Set `status: complete` in frontmatter when mutual understanding is achieved.
"Complete" means mutual understanding reached, not all fields populated.

## Output

Produces: `.docs/sketches/<slug>.md`
Passes to: blueprint

## References

- `references/sketch-template.md` -- YAML frontmatter schema, BUILD and FIX section templates

## Gotchas

- Ask user's instinct before presenting approaches. Anti-anchoring prevents first-option bias.
- Sketch is what/why, not how. No implementation details, no code, no file paths for changes.
- Blocking questions actually block. Do not proceed to blueprint with unresolved blockers.
- FIX mode must reference diagnose findings. Do not re-investigate from scratch.
- "Complete" means mutual understanding, not exhaustive fields. Empty sections are fine if intentional.

## Rationalization Red Flags

If you catch yourself thinking any of these, STOP and follow the sketch procedure:

1. "I already know what to build" — you might; the user might not share your assumptions
2. "Let me skip to blueprint, this is straightforward" — sketches catch wrong assumptions before they become wrong implementations
3. "I'll present my recommendation first, then ask" — anchoring. Ask instinct first.
4. "One approach is clearly best, no need to list alternatives" — OPTIONS-THEN-RECOMMEND. Always.
5. "The user seems impatient, I'll keep this brief" — brief is fine; skipping sections is not
6. "This is just a bug fix, it doesn't need a sketch" — diagnose decides that, not you
