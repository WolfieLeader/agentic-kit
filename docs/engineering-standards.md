# Engineering Standards for Skills & Agents

> Lessons learned from reviewing Superpowers, Compound Engineering, CC10X, Everything Claude Code, and Matt Pocock's skills. These standards apply to **every** skill and agent in the framework — not just a single skill. When building or modifying any skill, use this section as a checklist.

## File Size & Progressive Disclosure

| Rule | Why |
|------|-----|
| **SKILL.md must stay under 200 lines.** | Every line in SKILL.md is loaded on every invocation — it competes with the codebase context the model needs. CE keeps ce:brainstorm at ~200 lines; Superpowers keeps brainstorming at ~164 lines. CC10X's 462-line monolithic skill is an anti-pattern. |
| **Use `references/` for phase-specific detail.** | Reference files are loaded on demand via Read. A 6-section progression guide only needs to be in context during the dialogue phase, not during agent dispatch or handoff. Fewer, larger reference files beat many fragmented ones (fewer Read tool calls, less framing-token overhead). |
| **Agent files must stay under 120 lines.** | Agents run as subagents with their own context windows. Every line of agent instruction competes with the files the agent needs to read to do its job. ECC's code-explorer is ~60 lines. |
| **Don't load references for lightweight scope.** | If a skill has a lightweight fast-path, skip loading detailed reference files. The model can handle a brisk walkthrough from the main SKILL.md instructions alone. |
| **Skill descriptions must contain only triggering conditions — never workflow summaries.** | The description field (~200-400 chars) is loaded on every skill-list scan. If it summarizes the workflow ("dispatches agents, then produces a document"), the model may follow the description shortcut instead of reading the full SKILL.md body. Superpowers documented this as a known anti-pattern. Description should answer only: "When should this skill activate?" — not "What does it do internally?" |

## Hard Gates & Drift Prevention

| Rule | Why |
|------|-----|
| **Use `<HARD-GATE>` tags for non-negotiable rules.** | Superpowers' proven pattern. Claude treats `<HARD-GATE>` with higher compliance than prose "hard rules" or parenthetical labels. Use for: no code before design is complete, no recommendation before all approaches are shown, artifact must be written before handoff. |
| **Add a Rationalization Prevention table to every skill that has skippable phases.** | Inspired by CC10X. A table of "if you're thinking X, STOP — you're about to skip a step" creates a self-monitoring loop. Common rationalizations: "the answer is obvious," "this is too simple," "I already know what they want," "I'll combine steps to save time," "the user seems impatient so I'll skip ahead." |
| **Include violation examples for critical rules.** | A rule without a violation example is a guideline. "Never lead with your recommendation" is weak. Showing what violation looks like ("I recommend Approach A — here are the others for completeness...") makes the constraint checkable. CE and Superpowers don't do this; we should. |
| **`<HARD-GATE>` candidates include agent dispatch and section pacing — not just output rules.** | Round 1 identified 3 candidates: no code before design, no recommendation before all approaches, artifact before handoff. Round 2 experts unanimously added 2 more: (4) agent dispatch is mandatory for all scope tiers (the #1 rationalization is "this is too simple for agents"), (5) one section per message — the #1 behavioral drift across all reference frameworks. All 5 are `<HARD-GATE>` grade. |
| **Mandatory agent dispatch for lightweight scope is a deliberate tradeoff — monitor throughput.** | CE trims ceremony for lightweight work: "Search for the topic, check if something similar already exists, and move on" (line 99), and even skips document creation for brief alignment (line 186). Our framework mandates full agent dispatch for all tiers. The rationale is sound (30-second parallel Sonnet agents catch "already exists" before wasted design), but this is untested on throughput. If lightweight tasks consistently take >2 minutes in agent dispatch for <30 min of work, revisit. The escape hatch is not to skip agents — it's to make agents faster (tighter prompts, fewer files). |
| **State explicit instruction hierarchy when multiple files are in context.** | When SKILL.md + reference files + agent files are loaded together, the model needs to know which governs what. State it once: "SKILL.md governs the overall workflow. Reference files provide phase-specific detail. Agent files govern their own execution but receive subject and scope from the skill's dispatch prompts." Without this, conflicting instructions between files cause unpredictable behavior. |

## Interaction Discipline

| Rule | Why |
|------|-----|
| **One section per message — wait for response before advancing.** | The #1 behavioral drift in multi-section skills: the model presents Section 1 + Section 2 + Section 3 in one message, collapsing incremental confirmation into a rubber-stamp. Explicit instruction to STOP and wait after each section. "Brisk" (for lightweight) means shorter exchanges, not fewer sections. |
| **Cap clarifying questions at 2 before proceeding.** | Pre-dispatch or pre-planning clarification can loop forever on ambiguous requests. Max 2 questions, then proceed with best available understanding. Unbounded clarification delays the work without proportional improvement. |
| **Gate scope classification on user confirmation.** | If a skill classifies scope (lightweight/standard/deep), the user must explicitly confirm before the skill adjusts ceremony. Misclassification wastes time in both directions. Default-up when in doubt: over-classifying wastes 5-10 min; under-classifying risks missing critical constraints. |
| **Ask what the user is already thinking before presenting AI findings.** | CE's pattern (line 148 of ce:brainstorm). When opening a section, first ask "What's your current thinking about this?" THEN present agent findings as supplemental context. This surfaces hidden context the user has and prevents anchoring on the AI's framing before the developer has articulated their own. Skipping this turns the skill from a thinking partner into a form-filler. |
| **Sycophancy resistance — hold well-reasoned positions when user disagrees.** | When the user pushes back, evaluate whether their argument changes your reasoning. If it does, explain what changed your mind. If it doesn't, maintain your position and explain why. No reference framework handles this well — the default model behavior is to immediately agree. "I see your point, I'll go with your suggestion" when your reasoning is sound is sycophancy, not collaboration. Research shows explicit counter-instructions reduce sycophantic behavior by up to 69%. |
| **"Be a thinking partner" requires concrete behavioral anchors.** | "Be a thinking partner" is an aspiration, not a behavioral instruction. It must be grounded with checkable actions: (1) ask what the user is thinking before offering ideas, (2) voice disagreement and explain why, (3) challenge framing — "is this the right problem or a proxy for a more important one?", (4) bring your own perspective, not just extract requirements. Without anchors, "thinking partner" decays into "interviewer" by turn 20. |

## Agent Design

| Rule | Why |
|------|-----|
| **Set a file budget for exploration agents.** | Without a cap, agents can read 50+ files on a large monorepo. Max 15 files per investigation. If more context is needed, report findings so far and list what remains unexplored under Open Questions. CC10X uses a 10-file gate. Apply file budgets to ALL agents, including docs-explorer — not just code-focused agents. |
| **Handle agent failure explicitly.** | What if an agent returns nothing useful? What if external search fails? Skills must define what happens: note the gap in the summary, warn the user, proceed with adjusted expectations. Silent degradation is worse than acknowledged gaps. |
| **Require concrete search targets in agent dispatch prompts.** | Generic prompts ("investigate the codebase") produce generic findings. Every dispatch prompt must include (a) the specific area, (b) concrete search terms, and (c) what to look for. If the caller can't name 2 specific search targets, the request needs more clarification before dispatch. |
| **Agents must encode search procedures, not just responsibilities.** | Describing what to investigate ("find relevant patterns") is goal-level. Agents need procedure-level search methodology: (1) grep-first narrowing — use content search to pre-filter before reading files, (2) query echo — log the actual search terms used so the caller can assess coverage, (3) search count — report how many files were searched vs. read to show effort/selectivity. CE's `learnings-researcher` encodes a 7-step grep-first protocol with keyword extraction, category-based directory narrowing, and parallel search. CE's `repo-research-analyst` encodes scoped invocation with phase-selection logic. Our agents describe responsibilities well but leave the search method to the model's discretion, which produces inconsistent results. |
| **Agents must attribute findings to source.** | Distinguish between "the code-explorer found X in the codebase" and "based on my knowledge, Y is a common pattern." Presenting model assumptions with the authority of research findings erodes trust when those assumptions are wrong. |

## Artifact Design

| Rule | Why |
|------|-----|
| **Persist work during long dialogues, not just at the end.** | If a skill runs a multi-phase dialogue (6+ exchanges), write a draft artifact to disk before completion. Principle #1 (file-based handoff) has a gap between dialogue start and artifact write. If the user `/compact`s mid-dialogue, everything is lost. Writing a draft protects against this. |
| **Generate artifact identifiers early enough for draft persistence to work.** | If a skill writes mid-dialogue drafts (e.g., after Section 3), the artifact path must be deterministic at that point. Generate the slug (or any path identifier) in Phase 0/Phase 2 — not in the output phase. A draft persistence instruction that references a slug that hasn't been generated yet is non-executable. Validated bug: `/letsgo` Phase 3 referenced `<slug>` but slug generation was deferred to Phase 5. |
| **Split "Open Questions" into blocking vs. deferred.** | CE's pattern: "Resolve Before Planning" vs "Deferred to Planning." A flat Open Questions list doesn't tell the next skill whether it can proceed. Blocking questions must be answered first; deferred questions can be resolved during implementation. |
| **Add a completeness marker to artifacts.** | A crashed or interrupted skill may produce a half-written artifact. Downstream skills should be able to detect this. Either a YAML frontmatter field (`status: complete`) or a sentinel comment at the bottom. Prevents planning from a malformed brainstorm. |
| **Template compliance is a contract.** | When a skill writes an artifact from a template, follow the template exactly. Don't add sections, don't remove mandatory ones. The template is the contract between the producing skill and the consuming skill. |
| **Use YAML frontmatter for machine-parseable artifact metadata.** | Downstream skills need to read artifact metadata (scope tier, status, affected systems) programmatically. YAML frontmatter is machine-parseable; markdown bold fields (`**Scope:** standard`) require prose parsing. Move metadata to frontmatter: `title`, `date`, `scope`, `status`, `affected_systems`. The body remains markdown for human readability. |
| **Include a Research Context summary in brainstorm artifacts.** | The brainstorm captures decisions but not the research that informed them. If /plan reads brainstorm.md in a fresh session, it doesn't know what code-explorer or docs-explorer found. A brief "Research Context" section (between Affected Systems and Chosen Approach) summarizing key codebase findings — existing patterns, relevant files, prior work discovered — gives /plan crucial orientation without re-running research. |
| **Concrete test for the "no implementation details" boundary.** | The line between "architectural decision" and "implementation detail" is fuzzy. Use this test: "Would removing this detail change the brainstorm's conclusions?" If removing it wouldn't change any decision, it's an implementation detail and belongs in /plan. This prevents rationalized scope creep where technical details sneak into brainstorms under the "it's architectural" exception. |

## Stack Agnosticism

| Rule | Why |
|------|-----|
| **Never hardcode project-specific systems, services, or tech stack in skills or agents.** | Skills and agents must work regardless of which services the project uses or how many platforms it has. Reference CLAUDE.md or MAP.md for the project's system inventory. If the stack changes (add a service, remove a platform), only CLAUDE.md needs updating — not every skill file. |
| **Escalation heuristics should be principle-based, not list-based.** | "Escalate externally when the topic involves third-party services your team doesn't control" is agnostic. Listing specific services (Stripe, Supabase, etc.) is fragile. The principle covers future services automatically. |

## Anti-Anchoring

| Rule | Why |
|------|-----|
| **Present ALL options before recommending.** | This is the single most important behavioral rule in brainstorming and planning skills. Leading with a recommendation anchors the conversation — the developer evaluates the AI's pick instead of genuinely considering alternatives. The sequence is always: options → tradeoffs → recommendation. |
| **Require at least one non-obvious alternative.** | CE's pattern: inversion ("what if we did the opposite?"), constraint removal ("what if X weren't a factor?"), analogy ("how does another domain solve this?"). The first approaches that come to mind are variations on the same axis. Push for at least one that challenges the frame. |
| **Alternatives in artifacts must come from different axes.** | If the chosen approach is "build a new service," alternatives should include "use an existing library" or "extend an existing service" — not just "build a slightly different new service." Same-axis alternatives don't serve the anti-anchoring purpose. |

## Self-Review

| Rule | Why |
|------|-----|
| **Inline self-review is the minimum; independent review is required for standard/deep.** | The same model that wrote the output reviews it — structural bias. For lightweight scope, inline self-review with 5 checks is sufficient. For standard/deep scope, add an independent review step after artifact write: either dispatch a document-review subagent (CE's approach — runs the `document-review` skill post-write, auto-applies fixes, surfaces P0/P1 findings) or add an explicit user review gate (Superpowers' approach — "Artifact written to `<path>`. Please review and let me know if you want changes before we proceed." Wait for approval before handoff). Our current framework only has inline self-review, which is the weakest quality loop for a framework explicitly designed to resist drift. |
| **Placeholder bans have two tiers.** | **Hard ban** (auto-fix): "TBD", "TODO", "etc.", "similar", "and so on", "as needed". **Soft ban** (flag + replace with concrete alternative): "appropriate", "relevant", "necessary", "proper", "handle accordingly", "standard", "as described above". Principle: if an implementing agent reading the phrase would need to make a judgment call about what it means, it's a placeholder. |
| **Self-review must adopt an adversarial posture.** | "Your job is to find flaws, not confirm quality. Ask: if I were a developer reading this cold, what would confuse me?" Without this framing, the model defaults to confirming its own reasoning. |
| **Add a handoff readiness check.** | Beyond placeholders, consistency, scope, and completeness, add a 5th self-review check: "Could the next skill operate from this artifact alone, without any conversation context?" If /plan would need to invent product behavior, scope boundaries, or success criteria, the brainstorm is incomplete. This is the ultimate artifact self-sufficiency test. |

## Workflow Patterns

| Pattern | When to Apply | How |
|---------|--------------|-----|
| **Section lock with escape hatch** | Any skill with incremental section-by-section confirmation | Lock each section after user confirms. Unlock only when: (1) user explicitly requests revisiting, (2) a later section's findings invalidate an earlier decision (state the specific contradiction), or (3) agent research returns information that changes the premise. When unlocking, state what changed and which downstream sections need re-evaluation. |
| **"Already exists" fast-exit** | Any brainstorming or planning skill that dispatches research agents | If agents discover the feature already exists with adequate coverage, don't design it again. Present the finding and ask: (a) modify existing implementation, (b) extend it, or (c) confirm this is actually different work. If (a) or (b), reframe around the delta. |
| **Scope override handling** | Any skill with lightweight/standard/deep tier routing | When the user overrides the proposed scope tier: for an upgrade (e.g., lightweight→standard), re-dispatch agents with more specific prompts if initial findings were thin. For a downgrade (e.g., standard→lightweight), condense remaining steps but don't skip any — each step still gets addressed, just more briskly. |
| **Completed work as reference material** | Any agent that scans `.docs/plans/` | Completed brainstorms (those with a corresponding plan.md) ARE reference material from prior decisions. Only in-progress brainstorms (brainstorm.md without plan.md) should be treated as non-reference. Don't exclude an entire directory when only in-progress items should be excluded. |
| **Clear requirements fast-exit** | Any brainstorming skill where the user arrives with a fully formed design | If the user has already thought through the design and just needs it documented, don't force a full 6-section interrogation. Confirm understanding with a brief summary, then go straight to artifact creation. The lightweight tier helps but still requires all 6 sections — a "capture" mode for pre-formed designs would be even faster. |
| **Backward flow from consuming skill to producing skill** | Any skill pipeline (e.g., /plan consuming /letsgo output) | What if /plan discovers the brainstorm is insufficient? There must be a documented return mechanism: /plan flags what's missing and redirects to /letsgo with specific questions. CE has explicit "Resume ce:brainstorm" routing. Without backward flow, /plan either invents missing context (dangerous) or blocks without a path forward. |
| **User review gate before handoff** | Any skill that writes an artifact and hands off to the next skill | After writing the artifact and before presenting handoff options, pause for user review: "Artifact written to `<path>`. Please review and let me know if you want changes before we proceed." Wait for explicit approval. Superpowers does this at line 126 of brainstorming/SKILL.md. CE does this via a separate `document-review` skill at Phase 3.5. Without this gate, the handoff question ("proceed to /plan?") pressures acceptance of an unreviewed artifact. |
| **Build consumers before adding producer sophistication** | Any skill pipeline where the producer is built but the consumer is not | Build the consuming skill (/plan) before adding more sophistication to the producing skill (/letsgo). An unproven handoff contract is the biggest risk in a skill pipeline — the producer may write artifacts the consumer can't actually use. CE's strength is that ce:brainstorm and ce:plan are a working producer/consumer pair. Until both ends exist, the artifact contract is hypothetical. |
